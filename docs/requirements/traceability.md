# V1A normative requirements and verification matrix

## Purpose and authority

This is the canonical traceability record for Axiom V1A. It derives stable
requirements from the attached **Axiom V1A End-to-End Implementation Plan**
(`Plan`) and `crypto_bot_v1_codex_spec.md` Section 31.2 (`Spec`). The product
specification remains authoritative if wording here is incomplete or conflicts.
The plan's compatibility path, `docs/requirements-traceability.md`, links here.
Clause-by-clause plan coverage is maintained in
[source-coverage.md](source-coverage.md).

The current catalogue contains **381 requirement rows**: design/review contracts
in A0, implementation and measured acceptance in A1–A11, and cumulative release
proof under `RG`.

One ID represents one independently verifiable obligation. Repeated source
clauses map to the same canonical ID. A narrower locked-technology decision may
remain separate from the behavior that implements it only when the two rows have
different verification boundaries. Nested lists stay together only when they are
one atomic contract (for example, one order-book synchronization algorithm).
IDs are immutable. A removed or consolidated requirement is retained as
`Retired`, names every superseding ID in its row and in
[source coverage](source-coverage.md#retired-and-superseded-matrix-ids), and is
never reused. An active umbrella may not duplicate its atomic children.

## ID and status conventions

ID format: `AX-V1A-A<phase>-<class>-<sequence>`, where phase is `00` through
`11` and class is `SAF` (safety/security), `FUN` (functional), `OPS`
(operational), or `QLT` (quality/evidence). Release-wide gates use phase `RG`.

Allowed statuses are `Not started`, `In progress`, `Implemented`, `Verified`,
`Blocked`, and `Retired`. `Implemented` is not release evidence. Only `Verified`
means that the planned check passed and its immutable evidence is registered in
[V1A readiness](../releases/v1a-readiness.md). A0 through A6 are complete;
later phases and the V1A release remain incomplete.

Source abbreviations:

- `P§1`–`P§10`: sections of the attached V1A implementation plan.
- `S31.2/Ax`: the named V1A phase or release gate in specification Section 31.2.
- A source containing several repeated clauses is listed once on the matching
  canonical obligation.

Evidence entries below point to the readiness gate where the eventual immutable
artifact must be registered. `Pending` means evidence does not yet exist; it is
not a waiver and cannot satisfy a gate.

A0 is intentionally architecture-only. An A0 row may verify that a policy,
target, owner, SLI, test plan, and failure rule are defined and reviewed. It must
not require a later binary, runtime measurement, soak, restore drill, or browser
test. Those proofs use owning A1–A11 or `RG` IDs.

`AX-V1A-A01-QLT-013` is intentionally listed in the cross-cutting section. It is
a continuously enforced documentation obligation for behavior introduced by
later phases, with final evidence at the V1A release gate. Its `A01` identifier
is immutable, but it is not a phase-local A1 acceptance item and remains in
progress after A1 closes; this does not waive or retire the requirement.

## Traceability maintenance rules

1. Update a row to `Implemented` only when the implementation target exists and
   targeted checks have passed. Add the exact commit and test artifact to the
   matching readiness gate.
2. Update a row to `Verified` only after an independent acceptance review has
   inspected evidence at the same scope as the requirement.
3. A phase gate requires every ID owned by that phase to be `Verified`, its entry
   dependency to be verified, limitations recorded, and no safety/accounting/
   determinism blocker open.
4. Waivers never apply to production-order lockout, exact accounting,
   deterministic replay, ownership/fencing, or fail-closed behavior. Any allowed
   non-safety waiver records approver, rationale, expiry, and remediation issue in
   the readiness evidence bundle.
5. A changed requirement receives either an updated clarification that preserves
   its meaning or a new ID. Semantic replacement retires the old ID and records
   the successor; it never rewrites historical evidence.
